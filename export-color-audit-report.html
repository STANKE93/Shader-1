2. Color Pipeline Overview

Color Ramp Hex (sRGB)
  → THREE.Color (sRGB → linear)
  → Float32Array uniform
  → GLSL shader (linear math)
  → Pass 1: bgScene → rt1 (linear, no encoding)
  → Pass 2: bandsScene → rt2 (linear, no encoding)
  → Pass 3: halftoneScene → screen or export RT

LIVE:   outputColorSpace = SRGBColorSpace → Three.js injects linear→sRGB on final pass
PNG:    no outputColorSpace → FloatType RT → Float32Array → 4096-entry LUT → sRGB manually
VIDEO:  outputColorSpace = SRGBColorSpace → captureStream(0) captures sRGB canvas

4. Detailed Issue Descriptions

ISSUE 1: Live render targets use UnsignedByteType (8-bit clipping)

Severity: Medium  |  File: src/scene.js, lines 114–118

Problem: The live viewport's intermediate render targets (rt1, rt2) are created with the default UnsignedByteType (8-bit per channel). Layer 2 uses AdditiveBlending, which can produce values above 1.0 in linear space. These get clamped to 255 in the 8-bit RT, losing HDR highlight detail.

Meanwhile, the PNG and video export paths create their intermediate RTs with HalfFloatType, which preserves the full HDR range. This means bright, saturated areas where additive blending pushes channels above 1.0 will look different between live and export.

Current code:

const makeRT = (w, h) => new THREE.WebGLRenderTarget(w, h, {
    minFilter: THREE.LinearFilter,
    magFilter: THREE.LinearFilter,
    format:    THREE.RGBAFormat,
})

Fix: Add type: THREE.HalfFloatType to the live RTs to match the export precision:

const makeRT = (w, h) => new THREE.WebGLRenderTarget(w, h, {
    minFilter: THREE.LinearFilter,
    magFilter: THREE.LinearFilter,
    format:    THREE.RGBAFormat,
    type:      THREE.HalfFloatType,
})

Impact: Eliminates the primary remaining source of live-vs-export color difference. Minimal GPU cost on modern hardware.

ISSUE 2: rgbToHex skips linear-to-sRGB transfer

Severity: Low-Medium  |  File: src/controls/controls.js, line 1672

Problem: The rgbToHex helper converts linear RGB floats directly to hex bytes without applying the sRGB transfer function. This is called from stopsFromUniforms when reading current ramp data, causing two issues:

  • The CSS gradient preview bar in the ramp widget shows incorrect (brighter) colors because linear values are interpreted as sRGB.
  • If ramp stops are read via stopsFromUniforms and then written back via applyRamp, a double-linearization occurs: the already-linear value gets packed into hex as if it were sRGB, then THREE.Color applies sRGB→linear again, progressively darkening colors on each round-trip.

Current code:

function rgbToHex(r, g, b) {
    const h = v => Math.round(Math.min(Math.max(v, 0), 1) * 255)
                   .toString(16).padStart(2, '0')
    return `#${h(r)}${h(g)}${h(b)}`
}

Fix: Apply the standard sRGB transfer function before packing:

function rgbToHex(r, g, b) {
    const toSRGB = v => v <= 0.0031308
        ? v * 12.92
        : 1.055 * Math.pow(v, 1/2.4) - 0.055
    const h = v => Math.round(Math.min(Math.max(toSRGB(v), 0), 1) * 255)
                   .toString(16).padStart(2, '0')
    return `#${h(r)}${h(g)}${h(b)}`
}

INFO: Time discontinuity after video export

Severity: Informational  |  File: src/scene.js, line 434

After video export completes and the live loop resumes, the animation jumps because timeOffset was not adjusted to account for wall-clock time spent during export. Not a color issue, but a noticeable visual artifact.

5. What Was Verified Correct

  • sRGB LUT — 4096-entry IEC 61966-2-1 piecewise function, 12-bit quantization, error ≤ 0.4 output units
  • PNG export renderer — intentionally omits outputColorSpace; no double sRGB encoding
  • FloatType readback RT — Float32Array preserves full linear precision for PNG export
  • renderPasses color space switching — intermediate passes stay LinearSRGBColorSpace; final pass uses SRGBColorSpace only when rendering to screen (not to export RT)
  • Video frame capture — captureStream(0) + requestFrame() is frame-accurate at any FPS
  • preserveDrawingBuffer — set on both live and offscreen renderers
  • HalfFloatType export RTs — export intermediate RTs use HalfFloatType for HDR headroom
  • Halftone luminance — Rec. 709 coefficients correctly applied to linear-space input
  • THREE.Color management — ColorManagement.enabled = true by default; hex→linear conversions in ramp setup are correct

6. Recommended Priority

  1. Issue 1 (live RT type) — Primary remaining mismatch source. One-line fix, high impact.
  2. Issue 2 (rgbToHex) — Fixes CSS ramp preview accuracy and prevents potential color drift in preset/programmatic workflows.
